# Flomerce i18n — Handoff Doc for Phases A, B, C, D

**Audience:** the next engineer picking up Flomerce internationalization work.
**Goal:** ship four discrete improvements to the existing i18n stack without re-deriving any context.
**Last verified against code:** see file paths below — every claim here was checked against the real source, not memory.

---

## 1. TL;DR

Flomerce already has a working i18n pipeline: bundled English, R2-cached non-English locales, smart incremental regenerate via Microsoft Translator, namespace-scoped regenerate, cost-preview UI, and per-site shopper content translation (System B). What's missing is the polish layer that turns a "works" system into a "feels professional and stays cheap" system.

Build in this order:

1. **Phase B** (1 hr) — instant admin re-render after regenerate, no manual page reload.
2. **Phase A** (3 hr) — per-key re-translate + manual override sticky bit. Eliminates 90% of Force-Regenerate use cases.
3. **Phase C** (2 hr) — Cloudflare CDN cache (7-day TTL) with auto-purge on every write. Cuts Worker invocations ~95%.
4. **Phase D** (3 hr) — global Translation Memory in D1. Cuts Microsoft API spend 30–60%, scaling with merchant count.

Total: ~9 hours of focused work. Each phase ships independently.

## 2. System overview — two parallel translation systems

Flomerce runs **two separate i18n systems**. Don't confuse them; they have different storage, different cost models, different invalidation rules.

### System A — Platform UI strings
Static UI text on landing/dashboard/admin/wizard/storefront chrome (buttons, labels, navigation, etc.). Source = English JSON files in `frontend/src/shared/i18n/locales/en/*.json`.

- **English:** bundled into the JS bundle at build time. Zero backend calls.
- **All other locales:** generated by Microsoft Translator, stored as merged JSON in R2 (`locales/{lang}.json` + `locales/{lang}.hashes.json`), served by `/api/i18n/locale/:lang`.
- **Owner-controlled** via the i18n admin panel (Owner-only). Changes affect every merchant on the platform.

### System B — Per-merchant shopper content
Merchant-authored product names, descriptions, category names, etc. Source language varies per site (whatever the merchant types in).

- Stored per-site in D1 table `translation_cache(id, site_id, text_hash, source_lang, target_lang, translated_text, char_count, created_at)`.
- Content-addressed: cache key includes `text_hash = sha(source_text)`. When merchant edits text, the new hash is a cache miss → auto-translates on next shopper view.
- Browser layer: `sessionStorage` per `(siteId, target, hash)`.
- Each merchant pays for their own translations using their own Microsoft key (Growth+ feature).

**Phases A, B, C, D below are about System A unless explicitly stated.** System B is mostly self-managing because it's content-addressed.

## 4. Current cache stack — honest, layer by layer

| # | Layer | What it caches | Where it lives | Auto-purge on data change? |
|---|---|---|---|---|
| 1 | Browser HTTP cache | NOTHING (locale JSON) | Browser disk | N/A — explicitly disabled via `Cache-Control: no-store, no-cache, must-revalidate` in `cachedJson()` |
| 2 | Cloudflare edge cache | NOTHING (locale JSON) | CF edge | N/A — explicitly disabled via `CDN-Cache-Control: no-store` |
| 3 | SPA in-memory `Map` | Merged locale JSON per language | `localeCache` in `init.js` line 47 | ✅ Dies on tab close/reload |
| 4 | sessionStorage (System B only) | Per-product translated strings keyed by source hash | Browser tab | ✅ Content-addressed: edited source → new hash → cache miss |
| 5 | R2 storage | Generated System A locale JSON + hashes file | Cloudflare R2 (`STORAGE` binding) | ✅ Overwritten on regenerate; hash sidecar auto-detects deltas |
| 6 | D1 `translation_cache` (System B) | Per-site, per-product translations | D1 (`DB` binding) | ✅ Hash-based; new merchant text = new key = cache miss |

Today's posture: **"never edge-cache, always go to origin."** Safe but every page load is a Worker invocation. Phase C fixes this.

## 5. Known UX wart — the misnamed "Purge" button

In `I18nAdminPanel.jsx`, the "Purge" button calls `DELETE /api/admin/i18n/locale/:lang`. The backend (lines 591–597) does:

```js
await env.STORAGE.delete(r2Key(lang));
await env.STORAGE.delete(r2HashKey(lang));
```

This is a **destructive delete of the R2 file**, not a cache flush. The locale row disappears from the admin panel because there's no R2 file left to list. This was the only thing "purge" could mean before there was a CDN layer to purge.

**During Phase C, do this cleanup:**
- Rename existing button → `Delete & reset` (accurate label for what it does)
- Add a new `Purge edge cache` action that calls Cloudflare's purge API without touching R2
- Auto-purge edge cache after every successful regenerate / force-regenerate / per-key edit / manual override save (no manual click required for normal flow)

The two operations are now distinct and clearly labeled.

## 6. The 4 phases — detailed work plans

### Phase B — Live update without page refresh (1 hour, do first)

**Problem.** Today, after clicking Regenerate, the admin sees a success toast but the page still shows old strings until they hard-reload. Reason: i18next has the old catalog in its `localeCache` Map.

**Industry pattern.** i18next's documented `reloadResources()` API + `bindI18n: 'languageChanged loaded'` re-render trigger + `BroadcastChannel` for cross-tab sync. Used widely; trivial to ship.

**Files to touch.**
- `frontend/src/shared/i18n/init.js`:
  - Add `bindI18n: 'languageChanged loaded'` inside the `react: { ... }` options object.
  - Export a helper: `export async function refreshLocale(lang) { localeCache.delete(lang); await i18n.reloadResources(lang); }` — the `localeCache.delete(lang)` call is **critical**, otherwise reloadResources reads back the cached promise and gets stale data.
  - Set up a single `BroadcastChannel('flomerce-i18n')`. Listen for `{type: 'invalidate', lang}` messages and call `refreshLocale(lang)` on receipt.
- `frontend/src/platform/src/components/I18nAdminPanel.jsx`:
  - In the success branches of `regenerate()`, `forceRegenerate()`, `refreshAll()`, after `await loadLocales()`, also call `await refreshLocale(lang)` and `channel.postMessage({type:'invalidate', lang})`.

**Acceptance criteria.**
- Open admin in two tabs. Regenerate Hindi in tab 1. Switch to tab 2 → translations update without reload.
- Switch app language to Hindi. Regenerate. UI updates without manual reload.

**Gotchas.**
- BroadcastChannel is same-origin only. That's fine — admin and storefront both run on the same root domain in this app.
- Don't broadcast when receiving — guard against echo loops with a "this tab originated the change" boolean.

### Phase A — Per-key re-translate + manual override sticky bit (3 hours)

**Problem.** Today, "the Hindi translation of `admin.invoice.gstNote` is wrong but the English source is unchanged" forces a Force Regenerate of the whole catalog — wastes money + risks clobbering other good translations. There's also no way for the owner to lock in a hand-written translation.

**Industry pattern.** Shopify Translate & Adapt + WPML + Wix + Lokalise + Crowdin + Phrase ALL ship per-key re-translate and manual-override sticky bits. Auto-translate must skip any string a human has overridden.

**Schema.** New D1 table:

```sql
CREATE TABLE IF NOT EXISTS i18n_overrides (
  lang TEXT NOT NULL,
  ns TEXT NOT NULL,
  key_path TEXT NOT NULL,        -- dotted path like "admin.invoice.gstNote"
  value TEXT NOT NULL,           -- the manual translation
  manually_edited INTEGER NOT NULL DEFAULT 1,  -- 1 = sticky, 0 = auto-only
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL,
  PRIMARY KEY (lang, ns, key_path)
);
```

**Backend changes** in `backend/workers/platform/i18n-worker.js`:
1. New endpoint: `POST /api/admin/i18n/key/:lang/:ns/:key` — re-translate a single key. Reads English value from `EN_CATALOG`, calls `translateBatch(env, [enValue], lang)`, writes the result into the merged R2 catalog at the right dotted path, updates the hash entry. **Skips if there's a manual override row with `manually_edited=1`**.
2. New endpoint: `PUT /api/admin/i18n/override/:lang/:ns/:key` body `{ value }` — upserts into `i18n_overrides`, then patches the R2 catalog with the override value.
3. New endpoint: `DELETE /api/admin/i18n/override/:lang/:ns/:key` — clears the override; next regenerate will auto-translate this key.
4. **Modify `regenerateIncremental()`** to read `i18n_overrides` for the target lang up front and (a) skip those paths in the toTranslate set, (b) splice the override value into the final translation map.
5. After every key-level write, call `purgeLocaleCache(env, lang)` (Phase C helper).

**UI changes** in `I18nAdminPanel.jsx`:
- New "Edit Translations" view per locale: a paginated table of `(key, English source, current translation, override input, [Re-translate this key] button)`.
- Override input shows current override (if any) with a clear button.
- Per-row "Re-translate" button calls the per-key endpoint and updates the row inline.
- Add an `i18n.actionRetranslateKey` and ~10 related keys to `owner.json`.

**Acceptance criteria.**
- Click Re-translate on one key → only one row's translation changes; preview cost shows ~$0.0001.
- Type a manual override, save, then Force-Regenerate the whole locale → the overridden value is preserved.
- Clear the override → next regenerate auto-translates that key again.

**Gotchas.**
- Override values bypass `protectPlaceholders` because the human typed the final value. But you should sanity-check that `{{tokens}}` in the source still appear in the override — warn on mismatch (Shopify does this).
- When updating one key in the merged R2 catalog, do an atomic read-modify-write of the whole `{lang}.json` file. R2 has no partial-update API.

### Phase C — Cloudflare CDN cache (7-day TTL) + auto-purge (2 hours)

**Problem.** Every locale fetch goes all the way back to the Worker. Wasteful at scale.

**Solution.** 7-day edge cache + automatic purge on every write. Industry-standard pattern (Shopify uses ~5min TTL, but with reliable auto-purge there's no reason not to go longer).

**Headers** — change `cachedJson()` in `i18n-worker.js`:

```js
function cachedJson(request, body, { lang } = {}) {
  const cors = corsHeaders(request);
  const headers = {
    ...cors,
    'Content-Type': 'application/json; charset=utf-8',
    'Cache-Control': 'public, max-age=604800, stale-while-revalidate=86400',
    'CDN-Cache-Control': 'public, max-age=604800',
  };
  if (lang) {
    headers['Cache-Tag'] = `i18n,lang-${lang}`;
    // ETag = content hash of the body. Browser sends If-None-Match next visit.
    headers['ETag'] = `"${etagFor(body)}"`;
  }
  return new Response(JSON.stringify(body), { status: 200, headers });
}
```

Pass `{ lang }` from `handleI18nPublic` so the right cache-tag and ETag get attached. Also add `If-None-Match` handling at the top of `handleI18nPublic` — return `304` with empty body if the request's `If-None-Match` matches the current ETag (cheap because ETag derives from R2 object metadata; you can compute it from `obj.etag` or from `hashString(JSON.stringify(body))`).

**Cache-bust helper** — new file `backend/utils/cdn-cache.js`:

```js
export async function purgeLocaleCache(env, lang) {
  if (!env.CF_API_TOKEN || !env.CF_ZONE_ID) {
    console.warn('[cdn-cache] CF_API_TOKEN or CF_ZONE_ID missing; skipping purge');
    return { ok: false, reason: 'missing-credentials' };
  }
  const t0 = Date.now();
  try {
    const res = await fetch(
      `https://api.cloudflare.com/client/v4/zones/${env.CF_ZONE_ID}/purge_cache`,
      {
        method: 'POST',
        headers: {
          'Authorization': `Bearer ${env.CF_API_TOKEN}`,
          'Content-Type': 'application/json',
        },
        body: JSON.stringify({ tags: [`lang-${lang}`] }),
      },
    );
    const json = await res.json().catch(() => ({}));
    const ok = res.ok && json.success;
    console.log('[cdn-cache] purge', { lang, ok, status: res.status, latencyMs: Date.now() - t0, errors: json.errors });
    return { ok, status: res.status, body: json };
  } catch (e) {
    console.error('[cdn-cache] purge threw', { lang, latencyMs: Date.now() - t0, error: e.message });
    return { ok: false, reason: 'exception', error: e.message };
  }
}
```

**Note on Cache-Tag plan requirement.** Cache-Tag-based purge requires a Cloudflare Enterprise plan. If the account is on Free/Pro, switch to URL-based purge — replace the body with `{ files: [\`https://${env.PUBLIC_HOST}/api/i18n/locale/${lang}\`] }`. Either works; URL-based is fine for the small number of locales involved. Confirm the plan with the user before implementing.

**Wire purge into ALL write paths** — wrap the existing handlers so it's impossible to forget:

```js
async function afterLocaleWrite(env, lang) {
  // Single chokepoint. Every write path MUST call this.
  return purgeLocaleCache(env, lang);
}
```

Call sites (all in `i18n-worker.js`):
- After `regenerateIncremental()` succeeds in `POST /regenerate/:lang`
- After `regenerateIncremental({force:true})` succeeds in `POST /force-regenerate/:lang`
- Inside the `refresh-all` loop after each successful per-locale regen
- Inside the new per-key endpoint (Phase A)
- Inside the new manual-override PUT/DELETE endpoints (Phase A)
- Inside the existing DELETE handler (after the `STORAGE.delete()` calls)

**Required secrets** (use the environment-secrets skill):
- `CF_API_TOKEN` — Cloudflare API token with `Zone → Cache Purge` permission scoped to the zone. Create at <https://dash.cloudflare.com/profile/api-tokens>.
- `CF_ZONE_ID` — the zone ID, visible on the zone's overview page.
- `PUBLIC_HOST` — only if using URL-based purge instead of cache-tag-based. The host the i18n endpoint serves on (e.g. `flomerce.com`).

**Acceptance criteria.**
- First request to `/api/i18n/locale/hi` from a fresh browser hits the Worker (CF-Cache-Status: MISS).
- Second request from a different browser/IP returns `CF-Cache-Status: HIT` and identical body.
- Click Regenerate Hindi in admin → within ~3 seconds, requests from new browsers see the new translation (HIT but on the new content).
- ETag round-trip: send `If-None-Match: "<currentEtag>"` → get `304 Not Modified` with empty body.

**Gotchas the next agent must avoid.**
- **Do NOT use the deploy SHA as the cache key.** Any unrelated backend deploy would invalidate every locale. The cache key is the URL itself; the ETag is the content hash. That's the right separation.
- Cloudflare Cache API (`caches.default`) is per-datacenter — don't try to use it as a global store. The 7-day cache lives in CF's edge network via the response headers, not via Worker code.
- Cache API also only works on custom domains, not `*.workers.dev`. This is fine because the system already runs on a custom domain.
- Some CDNs ignore query strings in the cache key. Don't rely on `?v=hash` to bust cache; rely on the explicit purge call.
- If `purge_cache` fails (network blip, expired token), don't fail the user's request — the response should still succeed; the CDN will self-heal at the 7-day TTL.

### Phase D — Translation Memory (3 hours)

**Problem.** Common UI strings ("Add to cart", "Checkout", "Out of stock") get translated to Hindi over and over for every merchant. Multi-tenant platform → massive duplicate spend.

**Industry pattern.** TM (Translation Memory) is a core feature of every TMS: Lokalise, Crowdin, Phrase, Smartling, Transifex. Documented industry savings: 30–60% on machine-translation bills. For multi-tenant platforms with shared UI, real-world hit rate is 70–90%.

**Schema.** New D1 table:

```sql
CREATE TABLE IF NOT EXISTS translation_memory (
  hash TEXT PRIMARY KEY,             -- sha256(NFC(source) + '|' + targetLang)
  source_text TEXT NOT NULL,
  target_lang TEXT NOT NULL,
  translated_text TEXT NOT NULL,
  provider TEXT NOT NULL DEFAULT 'msft',
  created_at INTEGER NOT NULL,
  hit_count INTEGER NOT NULL DEFAULT 0
);
CREATE INDEX IF NOT EXISTS idx_tm_lang ON translation_memory(target_lang);
```

**Hash spec.**
```js
async function tmKey(sourceText, targetLang) {
  const normalized = String(sourceText).normalize('NFC');
  const data = new TextEncoder().encode(`${normalized}|${targetLang}`);
  const buf = await crypto.subtle.digest('SHA-256', data);
  return Array.from(new Uint8Array(buf)).map(b => b.toString(16).padStart(2, '0')).join('');
}
```

**Where to integrate.** Splice into `translateBatch()` in `backend/utils/translator.js` (NOT `translateCatalogIncremental` — keep TM transparent to all callers including System B):

```js
export async function translateBatch(env, texts, targetLang, sourceLang = 'en') {
  if (!Array.isArray(texts) || texts.length === 0) return [];
  if (!env.DB || sourceLang !== 'en') {
    // No TM table available, or non-English source — fall back to direct call.
    return _translateBatchUncached(env, texts, targetLang, sourceLang);
  }
  // 1. Compute hashes for every input string.
  const hashes = await Promise.all(texts.map(t => tmKey(t, targetLang)));
  // 2. Bulk lookup (chunk by ~100 to stay under D1 query size).
  const hits = await tmBulkLookup(env, hashes);   // { hash: translated_text }
  // 3. Build the miss list, preserving original order.
  const missIndices = [];
  const missTexts = [];
  for (let i = 0; i < texts.length; i++) {
    if (!hits[hashes[i]]) {
      missIndices.push(i);
      missTexts.push(texts[i]);
    }
  }
  // 4. Translate only the misses.
  const translated = missTexts.length
    ? await _translateBatchUncached(env, missTexts, targetLang, sourceLang)
    : [];
  // 5. Persist the new translations to TM (best-effort; don't block on failure).
  if (translated.length) {
    ctx?.waitUntil?.(tmBulkInsert(env, missIndices.map((idx, j) => ({
      hash: hashes[idx], source: texts[idx], targetLang, translated: translated[j],
    }))));
  }
  // 6. Fan out: every position in the output gets either a hit or a fresh translation.
  const out = new Array(texts.length);
  for (let i = 0; i < texts.length; i++) out[i] = hits[hashes[i]];
  for (let j = 0; j < missIndices.length; j++) out[missIndices[j]] = translated[j];
  // 7. Bump hit_count on the hits (background, don't block).
  const hitHashes = hashes.filter(h => hits[h]);
  if (hitHashes.length) ctx?.waitUntil?.(tmBumpHits(env, hitHashes));
  return out;
}
```

`_translateBatchUncached` is the current `translateBatch` body (Microsoft Translator HTTP call) renamed.

**Cost telemetry to add.** Track per-call stats: `{ requested: texts.length, tmHits: hitHashes.length, sentToProvider: missTexts.length, charsBilledAvoided, charsBilledSent }`. Surface in admin (new card: "TM efficiency: 73% hit rate, $X saved this month").

**Acceptance criteria.**
- First merchant signs up, generates Hindi → ~all keys are misses, all written to TM.
- Second merchant generates Hindi → most keys are hits, very few API calls. Verify `hit_count` increments.
- TM survives across sites — same hash, two sites reusing same translation.
- Edit one English source string → that single key becomes a miss (different hash), fresh translation, saved with new hash. Old TM row stays intact (immutable by design).

**Gotchas.**
- **Don't include glossary/manual-override state in the hash.** Keep the hash purely `(source, targetLang)`. Manual overrides are applied AFTER translation in Phase A's flow, so TM stays content-addressed and immutable.
- For System B (per-site shopper content), TM is still platform-wide because the hash is over `(source_text, target_lang)`. This is desirable: when one merchant's "Free shipping" text gets translated, every other merchant benefits. Quality risk is minimal because product-description text is rarely identical across merchants.
- D1 query size limits — chunk bulk lookups to ~100 hashes per query.
- Don't `INSERT OR REPLACE` on conflict — use `INSERT OR IGNORE`. TM is immutable; if the hash already exists, leave it.

## 11. Recommended commit hygiene

- Ship each phase as its own PR / task. They're independent.
- Phase B touches only `init.js` + `I18nAdminPanel.jsx` — tiny diff, easy review.
- Phase A adds one D1 table + 3 endpoints + an admin sub-view. Medium diff.
- Phase C touches `cachedJson()`, adds `cdn-cache.js`, wires `afterLocaleWrite()` into 5+ existing handlers, requires 2 new secrets. Test that purge works end-to-end before declaring done.
- Phase D touches `translator.js` (the most-used utility in the codebase) — write tests for the bulk-lookup / fan-out logic before shipping.

After each phase, run a code review using the code_review skill. Do not skip — the i18n pipeline already serves real merchants and a regression here is user-visible.
