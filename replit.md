# Flomerce SaaS Platform

## Overview
Flomerce is a multi-tenant SaaS platform designed for small businesses and entrepreneurs to create scalable e-commerce websites from templates, each hosted on a unique subdomain (e.g., `store-name.flomerce.com`). It provides efficient online storefronts, comprehensive sales capabilities, and brand-building tools through a Cloudflare-centric serverless architecture. The platform aims to be an accessible and powerful solution for establishing an online presence. Key features include dynamic content, subscription management, an admin panel with an iframe preview, product options, subcategories, multi-currency support, a robust order flow with cancellation and return systems, product reviews, inventory locations, storage usage tracking, admin shard management, built-in analytics, and global shipping configurations.

## System Architecture

### Core Design
Flomerce utilizes a shared shard-based D1 database architecture where multiple sites share Cloudflare D1 databases (shards) with `site_id`-based row isolation. A platform database (`env.DB`) stores global data (users, site metadata, subscriptions), while site-specific data (products, orders) resides in shard D1 databases, always filtered by `site_id`.

### Shared Shard Architecture
- **Shards Registry:** `shards` table in the platform DB tracks all shard databases.
- **Site Assignment:** New sites are assigned to an active shard.
- **Binding Pattern:** Shards are dynamically bound to workers.
- **DB Resolution:** Utilities resolve the correct shard DB.
- **Subscription Management:** Plan names are preserved on expiry, and subscription priority ensures correct trial reporting. Enterprise plan safeguards prevent accidental downgrades.

### Usage Tracking
- **Row-Level Tracking:** Estimated `row_size_bytes` contributes to `site_usage.d1_bytes_used`.
- **Correction Factor:** A per-shard `correction_factor` reconciles estimated usage with actual DB size.
- **R2 Tracking:** Managed via the `site_media` table.
- **Plan Limits:** Hard limits for Basic/Standard/Pro plans, with enterprise plans allowing overage.

### Plan-Based Feature Gating
- **Plan hierarchy:** Trial (full Pro access, 5 sites max, 14-day expiry) → Starter → Growth → Pro → Enterprise.
- **Count limits:** Sites (trial: 5), Staff (starter: 5, growth: 25), Inventory locations (starter: 2, growth: 50). Enforced in `sites-worker.js`, `site-admin-worker.js`, `inventory-locations-worker.js`.
- **Feature flags:** Reviews, Blog, Advanced SEO (incl. per-page/category/product SEO), and Coupons are available from Starter onwards (matches Shopify/Wix/Dukaan industry norms). Push Notifications (manual), Revenue analytics, Appointment Booking, and Remove "Powered by Flomerce" branding require Growth+. Push Notifications (automated) requires Pro+.
- **Backend enforcement:** `checkFeatureAccess(env, siteId, feature)` and `checkCountLimit(env, siteId, limitType)` in `backend/utils/usage-tracker.js`. Feature checks use `request.clone().json()` fallback to extract `siteId` from body when query param is missing. `GET /api/plan-limits` endpoint returns all limits for a site (incl. `appointmentBooking`, `removeBranding`). Storefront appointment endpoint (`POST /api/email/appointment`) gates on `appointmentBooking` when `siteId` is provided in the body. Removing branding is enforced storefront-side: the Footer renders "Powered by Flomerce" unless `settings.footer.hideBranding === true` AND `siteConfig.subscriptionPlan` is Growth+, so a downgraded merchant automatically loses the toggle's effect.
- **Frontend gating:** `FeatureGate` overlay component (`frontend/src/storefront/src/components/admin/FeatureGate.jsx`) wraps locked sections with upgrade prompt; normalization helpers are extracted to `frontend/src/storefront/src/utils/plan.js` (`normalizePlan`, `isPlanAtLeast`) so storefront components share the same plan logic as the admin. `AdminSidebar` shows lock icons and plan badges. `VisualCustomizer` gates the Book Appointment section (Growth+); Blog, Customer Reviews, and SEO subtabs are unlocked from Starter. The Footer editor (Edit Website → Footer tab) has a "Branding" card with a `Hide "Powered by Flomerce"` toggle gated at Growth+. Storefront `Navbar`, `NavbarModern`, `StoreLocations`, and `BookAppointmentPage` hide/redirect the appointment booking flow when the site plan is below Growth. `DashboardPage` enforces trial site creation limit and staff count limits with disabled buttons.
- **Trial expiry:** Enforced just-in-time by `reconcileSiteSubscription()` in `backend/workers/platform/sites-worker.js`, called on every site fetch. Sites with `subscription_plan='trial'` and a past `subscription_expires_at` are cleared (plan and expiry reset to NULL) and treated as `free` until the user upgrades. Active subscriptions and the `enterprise` admin override are preserved.
- **`subscription_plan`** is included in site info response and mapped as `siteConfig.subscriptionPlan` in `SiteContext.jsx`.

- **DEPLOY GOTCHA — editing a System A translation JSON requires BOTH a frontend AND a backend deploy.** The English source files under `frontend/src/shared/i18n/locales/en/` (`common.json`, `nav.json`, `landing.json`, `languages.json`) live in the frontend tree, but `backend/workers/platform/i18n-worker.js` imports them statically and bundles them into the worker at build time as `EN_CATALOG`. Three things follow:
  1. **The frontend deploy alone does nothing useful.** It only ships the new `t('keyName')` references. If the worker isn't redeployed, `EN_CATALOG` still has the OLD keys → `diffStaleness()` reports every cached non-English locale as `upToDate: true` → the **"Refresh all out-of-date" button never appears** in the admin Translations panel, and clicking per-language "Regenerate" produces a Hindi catalog that's missing the new keys (so the SPA falls back to the English literal even after Regenerate).
  2. **The worker MUST be redeployed for the new EN catalog to ship.** `cd backend && npx wrangler deploy`. After a successful backend deploy, verify with `curl https://flomerce.com/api/i18n/locale/en` (incognito) and confirm the new key is present. Only then will the admin staleness diff light up and Regenerate / Refresh-all produce a Hindi catalog containing the new keys.
  3. **CI pipelines that only watch `backend/**/*.js` paths will skip the redeploy** when the only edit is to a `.json` file under `frontend/src/shared/i18n/locales/en/`, because no `.js` file under `backend/` actually changed (the JSON import is resolved at wrangler bundle time, not at git-diff time). If you hit this in the future, the cheapest workaround is to make a trivial edit to `backend/workers/platform/i18n-worker.js` (e.g. tweak a comment) in the same commit so CI sees a `.js` change. The cleaner fix is to broaden the CI path filter to also trigger backend deploys when `frontend/src/shared/i18n/locales/en/**/*.json` changes — pick whichever your pipeline supports.
  4. **End-to-end staleness chain after a successful backend deploy:** new EN bundled → worker's `EN_CATALOG` fingerprint differs from each cached locale's stored fingerprint → admin Translations panel shows orange "X translations out of date" banner with "Refresh all out-of-date" button → click it → worker re-translates only the changed keys via Microsoft Translator → writes new JSON to R2 → calls `purgeLocaleCache()` to bust the 7-day Cloudflare edge cache (requires `CF_API_TOKEN` wrangler secret + `CF_ZONE_ID` env var, both already configured) → next browser request fetches fresh catalog from origin → SPA hard-refresh required to drop i18next's in-memory resource store → user sees translated text. **Every step must succeed; the chain is fragile precisely because each layer caches aggressively.** Standalone "translate and ship" workflows are intentionally avoided — the staleness diff is the source of truth and humans gate the spend.

Purges are debounced per-site by default: the first call for a `siteId` opens a fixed 30s window (configurable via `env.PURGE_DEBOUNCE_MS`), and every subsequent purge call within that window for the same site is coalesced — its `types` and `resourceIds` are merged into the same pending entry and a single Cloudflare API call fires when the window expires. The window is NOT reset by later edits (bounds `ctx.waitUntil` to ≤ one window). Coalesce state is in-memory per Worker isolate, so concurrent admin requests on different isolates each run their own debouncer (worst case: one purge per isolate per window per site — still dramatically better than per-edit). Pass `{ immediate: true }` as the 5th arg to `purgeStorefrontCache(env, siteId, types, resourceIds, { immediate: true })` to bypass debouncing for events that must purge instantly (site deletion, manual admin purge action). Stale risk on isolate eviction during the 30s window: at most one merchant edit's purge is dropped; affected URLs stay stale until next edit triggers a purge or until the 7-day TTL expires.

  - **Cron dispatcher mirror rule (Apr 27, 2026 — second pass):** the cron strings inside the `scheduled(event, env, ctx)` `if (cron === '...')` chain in `backend/workers/index.js` MUST match `wrangler.toml` `[triggers].crons` exactly. Earlier today the cadence was bumped from `'0 * * * *'` to `'*/15 * * * *'` in `wrangler.toml` but the dispatcher still matched the old hourly string, so for every 15-minute fire Cloudflare delivered, the dispatcher saw an unknown cron and fell through to the old "fallback" branch — which silently ran `cleanupExpiredData` + `cleanupOrphanMedia` instead of the abandoned-cart sweep. Net effect: zero abandoned-cart emails sent in production, and cleanup queries running 96×/day instead of 1×/day. Fixed by (a) updating the dispatcher string to `'*/15 * * * *'`, and (b) replacing the unknown-cron fallback with a `console.error` no-op so future cron-key mismatches fail loudly instead of being masked by a "helpful" fallback. If you change the cron string in either file again, change both — there is now a comment in `scheduled()` calling this out.
  - **Google sign-in cart merge:** `handleCustomerGoogleLogin` (`backend/workers/storefront/customer-auth-worker.js`) accepts a `sessionId` (body field or `X-Session-ID` header), then calls `mergeCarts(env, siteId, customer.id, sessionId)` after the customer is resolved/created. Without this the guest cart stays attached to `session_id` with `user_id = NULL`, and the reminder job (which requires `user_id IS NOT NULL`) skips it. Frontend `googleLogin()` in `services/authService.js` passes `sessionId: getSessionId()`. Failures are caught and logged — they never block sign-in.
  - **Observability:** `processAbandonedCartReminders` logs a one-time `[AbandonedCart] BREVO_API_KEY/FROM_EMAIL not configured` warning at run start (so an unconfigured Brevo doesn't fail silently per cart), per-site stats `[AbandonedCart] site=<id> ... candidates=N sent=email:N,wa:N skipped=backoff,recent_order,no_contact,no_customer,empty_items,parse_error,no_channel_dispatched,max_reminders_reached cfg=...`, and a final run summary `[AbandonedCart] Run complete — sites=N feature_disabled=N no_settings=N with_candidates=N candidates=N sent=email:N,wa:N skipped={...} errors=N`. The split between `feature_disabled` (merchant turned the toggle off) and `no_settings` (settings row missing entirely) lets operators distinguish intentional opt-out from misconfiguration. `max_reminders_reached` is computed from a separate `COUNT(*)` query because the candidate-selection SQL filters those carts out — without the count, a quiet log on a healthy site would be indistinguishable from a broken job. Scan production logs with `fetch_deployment_logs` filtered on `(?i)abandonedcart` to verify the hourly job is firing and to see why specific carts were skipped.
  - **Self-healing schema (Apr 27, 2026):** `ensureAbandonedCartColumns(env, db, siteId)` is called once per site at the top of every per-site iteration in `processAbandonedCartReminders` and adds five columns if missing — `carts.reminder_sent_at`, `carts.reminder_count`, `carts.language`, `site_customers.preferred_lang`, `orders.placed_in_language`. Each `ALTER TABLE` is wrapped in its own try/catch (no-op when column exists). Critically, the function is gated by a module-level `Set` (`abandonedCartHealedShards`) keyed on `shard:<shard_id>` (resolved from `sites.shard_id`) so the DDL fires **once per shard per Worker isolate**, not on every cron iteration — this avoids DB lock contention on the every-15-minute sweep. Without this self-heal, older site shards that pre-date the language migrations silently throw inside the per-site try/catch, the cart sweep sees 0 candidates / 0 sent for those shards, and there is no other clear signal that something is wrong. The 30-minute delay option (`value="0.5"` in the admin dropdown) flows through the candidate query as `datetime('now', '-' || 0.5 || ' hours')` — SQLite `datetime` accepts decimal hour modifiers, so no other code change is required to support sub-hour delays beyond the matching cron cadence above.

### Production vs Local Development
- **Always plan and reason for production first.** This codebase runs on Cloudflare Pages + Workers + D1 + R2. The local Wrangler dev mode is a Miniflare simulation and does NOT replicate production behavior for: edge cache (Cloudflare CDN cache for public endpoints), custom domain SSL provisioning, scheduled (cron) triggers, D1 sharding routing in production, R2 region pinning, and rate limits.
- **Code or DB changes don't reach the live site until both the backend Worker and frontend Pages are redeployed.** Local restarts only affect the Miniflare simulation.
- **For production debugging,** use `fetch_deployment_logs` (Worker logs) rather than relying on local console output.
